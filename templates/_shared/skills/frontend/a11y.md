# Accessibility (a11y)

Rules and patterns for building accessible interfaces. Apply on top of
`react.md` and `conventions.md`.

## Rules

- **Semantic HTML first.** Use `<button>`, `<a>`, `<nav>`, `<main>`,
  `<section>`, `<dialog>` — not `<div onClick>`. Semantic elements get keyboard
  and screen-reader support for free.
- **Every interactive element has an accessible name.** Buttons have visible
  text or `aria-label`. Inputs have `<label htmlFor>`. Images have `alt` (use
  `alt=""` for decorative images).
- **Keyboard navigable.** Every interactive element must be reachable via Tab
  and operable via Enter/Space. Custom widgets need explicit `tabIndex`,
  `onKeyDown`, and ARIA roles.
- **Visible focus indicators.** Never `outline: none` without a replacement. Use
  `focus-visible:ring-2` (Tailwind) or equivalent for keyboard-only focus.
- **Color is never the only signal.** Pair color with icons, text, or patterns
  for error states, status badges, and charts.
- **WCAG AA contrast.** Text must have 4.5:1 contrast ratio (3:1 for large
  text). Use tooling to verify.
- **ARIA as a last resort.** If you can achieve the behavior with HTML alone, do
  it. Add `aria-*` attributes only when native semantics are insufficient.
- **Focus management on route changes.** Move focus to the main content or
  heading when navigating via SPA router.
- **Announce dynamic content.** Use `aria-live="polite"` for toast
  notifications, form success messages, and async status updates.
- **Modals trap focus.** When a dialog/modal is open, Tab must cycle within it.
  Escape must close it. Focus returns to the trigger on close.

## Patterns

### Accessible button vs div

```tsx
// ❌ Non-semantic — no keyboard support, no role
<div onClick={handleClick} className="btn">
  Save
</div>

// ✅ Semantic button — keyboard, screen reader, and focus for free
<button onClick={handleClick} className="btn">
  Save
</button>
```

### Icon button with accessible name

```tsx
<button aria-label="Close dialog" onClick={onClose}>
  <XIcon aria-hidden="true" />
</button>
```

### Form with proper labels

```tsx
<div>
  <label htmlFor="email">Email address</label>
  <input
    id="email"
    type="email"
    aria-describedby="email-error"
    aria-invalid={!!errors.email}
    {...register("email")}
  />
  {errors.email && (
    <p id="email-error" role="alert">
      {errors.email.message}
    </p>
  )}
</div>
```

### Live region for async feedback

```tsx
<div aria-live="polite" aria-atomic="true">
  {saveStatus === "success" && <p>Changes saved successfully.</p>}
  {saveStatus === "error" && <p>Failed to save. Please try again.</p>}
</div>
```

### Skip navigation link

```tsx
// At the very top of the layout
<a
  href="#main-content"
  className="sr-only focus:not-sr-only focus:absolute focus:z-50 focus:p-4"
>
  Skip to main content
</a>;

{
  /* ... nav ... */
}

<main id="main-content" tabIndex={-1}>
  {children}
</main>;
```

### Focus trap in modal

```tsx
import { useEffect, useRef } from "react";

export function Modal({ isOpen, onClose, children }: ModalProps) {
  const dialogRef = useRef<HTMLDialogElement>(null);

  useEffect(() => {
    const dialog = dialogRef.current;
    if (!dialog) return;

    if (isOpen) {
      dialog.showModal(); // native focus trap
    } else {
      dialog.close();
    }
  }, [isOpen]);

  return (
    <dialog ref={dialogRef} onClose={onClose}>
      {children}
      <button onClick={onClose}>Close</button>
    </dialog>
  );
}
```

## Anti-patterns

```tsx
// ❌ div as button — no keyboard support
<div onClick={toggle} className="toggle">Toggle</div>

// ✅ Proper button
<button onClick={toggle}>Toggle</button>
```

```tsx
// ❌ Removing focus outline with no replacement
button:focus {
  outline: none;
}

// ✅ Custom focus indicator
button:focus-visible {
  outline: 2px solid var(--focus-ring);
  outline-offset: 2px;
}
```

```tsx
// ❌ Image without alt
<img src="/logo.png" />

// ✅ Informative alt
<img src="/logo.png" alt="Acme Corp logo" />

// ✅ Decorative image
<img src="/divider.png" alt="" role="presentation" />
```

```tsx
// ❌ Color-only error indicator
<input style={{ borderColor: hasError ? "red" : "gray" }} />

// ✅ Color + icon + text
<input aria-invalid={hasError} />
{hasError && (
  <p role="alert">
    <WarningIcon aria-hidden /> This field is required
  </p>
)}
```

## Checklist

- [ ] All interactive elements use semantic HTML (`button`, `a`, `input`)
- [ ] Every input has a visible `<label>` or `aria-label`
- [ ] Every image has `alt` (descriptive or `alt=""` for decorative)
- [ ] Keyboard navigation works for all interactive elements
- [ ] Visible focus indicators on all focusable elements
- [ ] Color is not the only way to convey information
- [ ] Contrast meets WCAG AA (4.5:1 text, 3:1 large text)
- [ ] Dynamic content uses `aria-live` regions
- [ ] Modals trap focus and return it on close
- [ ] Skip-navigation link present in layouts
- [ ] `aria-invalid` and `aria-describedby` used on form errors
