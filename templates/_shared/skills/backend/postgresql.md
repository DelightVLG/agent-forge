# PostgreSQL

Rules and patterns for working with PostgreSQL. Apply on top of `conventions.md`
and `db-migrations.md`.

## Rules

- **Design schemas in third normal form (3NF) as the starting point.**
  Denormalize only when you have measured query performance problems — not
  preemptively.
- **Use appropriate data types.** `uuid` or `text` for IDs (not `serial` for
  distributed systems). `timestamptz` for dates (never `timestamp` without
  timezone). `text` over `varchar` unless a hard length constraint exists.
  `numeric` for money (never `float`).
- **Every table has a primary key.** Prefer `uuid` generated by the application
  (`gen_random_uuid()` or `cuid()`). Use composite PKs only for join tables.
- **Foreign keys always have indexes.** PostgreSQL does not auto-index FK
  columns. Missing FK indexes cause slow `DELETE` cascades and join performance
  issues.
- **Use constraints liberally.** `NOT NULL` by default. `UNIQUE` for business
  keys. `CHECK` for domain rules. Constraints are the last line of defense.
- **Write idempotent migrations.** Use `IF NOT EXISTS` / `IF EXISTS` in DDL.
  This prevents failures on re-runs and simplifies rollback.
- **Use `EXPLAIN ANALYZE` to validate query performance.** Check for sequential
  scans on large tables, nested loops on big joins, and high row estimates vs
  actuals.
- **Connection pooling is mandatory in production.** Use PgBouncer in
  `transaction` mode or the driver's built-in pool. Set `max` connections to
  `(CPU cores * 2) + disk count` on the DB server.
- **Use transactions appropriately.** Default isolation level (`READ COMMITTED`)
  is correct for most cases. Use `SERIALIZABLE` only for financial operations.
  Keep transactions short — never hold a transaction open during I/O to external
  services.
- **Backup and test restores.** Automated backups are useless if you never test
  a restore. Verify backup integrity on a schedule.

## Patterns

### Table design

```sql
CREATE TABLE users (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  email       text NOT NULL UNIQUE,
  name        text NOT NULL,
  role        text NOT NULL DEFAULT 'user' CHECK (role IN ('user', 'admin', 'editor')),
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE posts (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  title       text NOT NULL,
  content     text,
  published   boolean NOT NULL DEFAULT false,
  author_id   uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at  timestamptz NOT NULL DEFAULT now()
);

-- FK index — PostgreSQL does not create this automatically
CREATE INDEX idx_posts_author_id ON posts (author_id);
```

### Index strategies

```sql
-- B-tree: equality and range queries (default)
CREATE INDEX idx_users_email ON users (email);

-- Partial index: only index rows that matter
CREATE INDEX idx_posts_published ON posts (created_at DESC)
  WHERE published = true;

-- Covering index: includes extra columns to enable index-only scans
CREATE INDEX idx_posts_author_covering ON posts (author_id)
  INCLUDE (title, created_at);

-- GIN index: full-text search, JSONB containment
CREATE INDEX idx_posts_content_search ON posts
  USING gin(to_tsvector('english', content));

-- Concurrent index creation for large tables (no table lock)
CREATE INDEX CONCURRENTLY idx_orders_user_id ON orders (user_id);
```

### Query optimization with EXPLAIN

```sql
-- Always check query plans for critical queries
EXPLAIN ANALYZE
SELECT u.name, COUNT(p.id) AS post_count
FROM users u
LEFT JOIN posts p ON p.author_id = u.id AND p.published = true
GROUP BY u.id
ORDER BY post_count DESC
LIMIT 10;

-- Look for:
-- ✅ Index Scan / Index Only Scan
-- ⚠️  Seq Scan on large tables (may need an index)
-- ⚠️  Nested Loop on large datasets (may need Hash Join)
-- ❌ Rows estimated vs actual differ by 10x+ (stale statistics → ANALYZE)
```

### Window functions

```sql
-- Rank users by post count within each role
SELECT
  name,
  role,
  COUNT(p.id) AS post_count,
  RANK() OVER (PARTITION BY role ORDER BY COUNT(p.id) DESC) AS rank
FROM users u
LEFT JOIN posts p ON p.author_id = u.id
GROUP BY u.id;
```

### CTE for complex queries

```sql
-- Monthly active users with running total
WITH monthly AS (
  SELECT
    date_trunc('month', created_at) AS month,
    COUNT(DISTINCT author_id) AS active_users
  FROM posts
  WHERE created_at >= now() - interval '12 months'
  GROUP BY 1
)
SELECT
  month,
  active_users,
  SUM(active_users) OVER (ORDER BY month) AS cumulative
FROM monthly
ORDER BY month;
```

### JSONB operations

```sql
-- Store flexible metadata
ALTER TABLE users ADD COLUMN metadata jsonb DEFAULT '{}';

-- Query JSONB fields
SELECT * FROM users WHERE metadata->>'plan' = 'premium';
SELECT * FROM users WHERE metadata @> '{"features": ["beta"]}';

-- GIN index for JSONB containment queries
CREATE INDEX idx_users_metadata ON users USING gin(metadata);
```

## Anti-patterns

```sql
-- ❌ timestamp without timezone — ambiguous across time zones
created_at timestamp NOT NULL DEFAULT now()

-- ✅ Always use timestamptz
created_at timestamptz NOT NULL DEFAULT now()
```

```sql
-- ❌ No index on foreign key
CREATE TABLE posts (
  author_id uuid REFERENCES users(id)
  -- no index!
);

-- ✅ Always index foreign keys
CREATE INDEX idx_posts_author_id ON posts (author_id);
```

```sql
-- ❌ Using float for money — precision loss
price float NOT NULL

-- ✅ Use numeric for exact decimal values
price numeric(10, 2) NOT NULL
```

```sql
-- ❌ Long-running transaction holding locks
BEGIN;
SELECT * FROM orders WHERE id = 1 FOR UPDATE;
-- ... call external payment API (takes 5 seconds) ...
UPDATE orders SET status = 'paid' WHERE id = 1;
COMMIT;

-- ✅ Minimize transaction scope
-- Call external API first, then wrap only the DB update in a transaction
```

```sql
-- ❌ SELECT * in application queries
SELECT * FROM users;

-- ✅ Select only needed columns
SELECT id, email, name FROM users;
```

## Checklist

- [ ] Tables start in 3NF; denormalization is measured and justified
- [ ] `timestamptz` used for all date/time columns (not `timestamp`)
- [ ] `numeric` used for money (not `float` or `double`)
- [ ] Every foreign key column has an index
- [ ] Constraints applied: `NOT NULL`, `UNIQUE`, `CHECK` where appropriate
- [ ] Critical queries verified with `EXPLAIN ANALYZE`
- [ ] Connection pooling configured (PgBouncer or driver pool)
- [ ] Migrations are idempotent (`IF NOT EXISTS` / `IF EXISTS`)
- [ ] Transactions are short — no external I/O inside transactions
- [ ] Backups tested with restore verification
